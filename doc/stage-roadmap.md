# 阶段路线图

## 文档目的

这份文档只做三件事：

- 说明当前项目处在哪个阶段
- 说明每个阶段的目标、范围和完成标准
- 作为阶段提交与后续推进的对照表

不在这里重复大而全的方案设计，详细背景仍以 `doc/project-plan-v0.md` 为准。

## 当前总体判断

当前主线已经从“前置验证”切换到“知识问答 MVP 实现”。

当前推进原则：

- 每完成一个小阶段并验证通过，优先先提交一版
- 不让 OpenClaw、发消息授权、向量检索阻塞主线
- 先把“文档整合 -> 检索 -> 问答”这条最小闭环做通

## 阶段 0：项目基建与飞书能力验证

### 目标

完成项目最小骨架搭建，确认飞书文档能力可用。

### 范围

- 仓库初始化
- 依赖与环境模板
- 基础 FastAPI 入口
- `lark-cli` 配置与授权
- 飞书文档搜索与抓取验证

### 完成标准

- 项目骨架可运行
- `docs +search` 可用
- `docs +fetch` 可用
- 基础配置项齐全

### 当前状态

- 已完成

## 阶段 1：文档采集与本地检索 MVP

### 目标

把飞书文档拉取到本地，并形成最小可检索能力。

### 范围

- `lark-cli` 搜索与抓取封装
- 文档落盘
- Markdown 切片
- BM25 本地检索
- FastAPI `/retrieve` 接口

### 完成标准

- 文档可抓取到本地 `data/raw_docs/`
- 本地文档可切片
- 本地文档可 BM25 检索
- `/retrieve` 返回可用命中结果
- 已完成实际验证

### 当前状态

- 已完成

### 本阶段产出

- `backend/clients/`
- `backend/ingestion/`
- `backend/models/`
- `backend/retrieval/`
- `backend/services/retrieval_service.py`
- `backend/api/routes/retrieval.py`
- `scripts/fetch_lark_doc.py`
- `scripts/search_local_docs.py`

## 阶段 2：问答接口 MVP

### 目标

在本地检索基础上接入真实大模型，形成最小问答闭环。

### 范围

- 检索结果组装
- Prompt 构造
- 调用 OpenAI 兼容大模型接口
- FastAPI `/answer` 接口
- 返回答案与证据片段

### 完成标准

- `/answer` 可用
- 输入问题后能返回答案
- 返回结果带证据来源或命中片段
- 本地 `.env` 中的 LLM 配置可正常工作

### 当前状态

- 已完成

### 当前进展

- 已完成 `/answer` 接口骨架
- 已串起“本地检索 -> Prompt 组装 -> LLM 客户端调用”主链路
- 已完成不依赖真实模型的接口级验证
- 已完成真实 MiniMax 配置下的联调验证
- 已处理模型返回 `<think>` 内容泄露问题

### 当前阶段风险

- `LLM_MODEL` 需要填写真实可调用的模型 ID，而不是展示名
- 若模型调用失败，先保留检索结果返回，不阻塞接口结构落地

## 阶段 3：检索增强

### 目标

在 BM25 之外补上语义检索或重排，提高问答稳定性。

### 范围

- Embedding 接入
- 向量索引
- 可选 rerank
- 与现有 BM25 形成混合检索

### 完成标准

- 至少形成“关键词检索 + 语义检索”的组合
- 检索结果质量相比纯 BM25 有可解释提升

### 当前状态

- 待开始

### 说明

这一阶段重要，但不应阻塞阶段 2。

---

## 阶段 5：变更感知与主动分发

### 目标

实现赛题差异化能力：文档变化后，自动分析并主动推送给相关人。

### 范围

- 文档变更检测
- 差异摘要
- 影响对象识别
- 主动通知

### 完成标准

- 能识别文档更新
- 能生成可读的变化摘要
- 能输出候选影响对象
- 能通过飞书侧完成通知

### 当前状态

- 待开始
