# 阶段路线图

## 文档目的

这份文档只做三件事：

- 说明当前项目处在哪个阶段
- 说明每个阶段的目标、范围和完成标准
- 作为阶段提交与后续推进的对照表

不在这里重复大而全的方案设计，详细背景仍以 `doc/project-plan-v0.md` 为准。

## 当前总体判断

当前主线已经从“后台能力验证”切换到“飞书原生 Demo 收口”，且 P0 / P1 核心链路已完成真实飞书验证。

当前推进原则：

- 不再只追求后台 API 可用，优先补齐“飞书里能问、飞书里能收到主动推送”的演示体验
- OpenClaw 定位为可选 HTTP Skill / 订阅入口，不作为主链路阻塞项
- 变更分发仍是项目差异化亮点，但必须至少有一种主动触发方式可演示
- 代码提交按核心里程碑收口，不再每个小验证都单独提交

### 赛题校准结论

`doc/subject.txt` 明确要求的是“办公场景驱动的智能知识助手”，不仅要精准问答，还要主动分发与推送，并且可运行 Demo 至少包含一种主动触发方式。

当前项目方向没有偏：飞书文档整合、基于证据的问答、文档变更感知、变更摘要和定向分发均贴合赛题。当前已完成飞书 Bot 问答和真实主动推送链路验证，下一阶段重点转为案例、录屏与效果验证材料收口。

## 阶段 2：问答接口 MVP

### 目标

在本地检索基础上接入真实大模型，形成最小问答闭环。

### 范围

- 检索结果组装
- Prompt 构造
- 调用 OpenAI 兼容大模型接口
- FastAPI `/answer` 接口
- 返回答案与证据片段

### 完成标准

- `/answer` 可用
- 输入问题后能返回答案
- 返回结果带证据来源或命中片段
- 本地 `.env` 中的 LLM 配置可正常工作

### 当前状态

- 已完成

### 当前进展

- 已完成 `/answer` 接口骨架
- 已串起“本地检索 -> Prompt 组装 -> LLM 客户端调用”主链路
- 已完成不依赖真实模型的接口级验证
- 已完成真实 MiniMax 配置下的联调验证
- 已处理模型返回 `<think>` 内容泄露问题

### 当前阶段风险

- `LLM_MODEL` 需要填写真实可调用的模型 ID，而不是展示名
- 若模型调用失败，先保留检索结果返回，不阻塞接口结构落地

## 阶段 5：变更感知与主动分发

### 目标

实现赛题差异化能力：文档变化后，自动分析并主动推送给相关人。

### 范围

- 文档变更检测
- 差异摘要
- 影响对象识别
- 主动通知

### 完成标准

- 能识别文档更新
- 能生成可读的变化摘要
- 能输出候选影响对象
- 能通过飞书侧完成通知

### 当前状态

- 已完成核心链路验证（真实主动推送已验证）
- 已补 OpenClaw 弱对接最小入口：`/api/openclaw/query` / `/api/openclaw/subscribe` + `openclaw-skills/enterprise-knowledge/SKILL.md`
- 已验证个人 Wiki 文档链路：抓取 -> 检索 -> 问答 -> baseline -> 修改文档 -> `changed=1` -> 变更摘要 -> dry-run 分发命令 -> `--watch --send` 真实推送到测试群

- **5/1 为 Phase 2 开始的硬死线**
- 即使阶段 3b 仍有尾项，也必须按时切入这一阶段
- 当前已落最小骨架：`state.py` / `watcher.py` / `differ.py` / `impact.py` / `dispatcher.py`
- 当前下一步重点不是继续扩后台能力，而是补足案例、效果验证与录屏材料，让 watcher 的 `--watch` / `--send` 以稳定演示脚本形式呈现
